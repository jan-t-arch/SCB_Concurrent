Aggregate Premium Maximum Check
總保費上限檢查

本頁解釋當客戶同日投保多份保單時,系統如何加總所有保單保費(轉 HKD)再與上限比較,判斷是否超標。分三種情況(單筆 / 每月 / 混合),每種計法不同。選擇 scenario、加入保單、修改數值,即可查看實時逐步計算。

⚠ 最易混淆之處 / The counter-intuitive part

保單清單 / Policies

上限(可編輯) / Maximum (editable)

此上限為可編輯 sample,方便測試 boundary。實際限額(可能與 VC / Super VC 規則相關)待確認。

逐步計算 / Step-by-step

✓ 已確認假設 / Confirmed assumptions

  • 所有金額轉為 HKD 後才加總。
  • 當前申請自身保費計入總數。
  • 手動加入、系統搜尋、當前申請同屬一個客戶(client-level 加總 / 供款能力式檢查)。
  • 上限採用嚴格大於:總數 > 上限 才算超標(等於上限 = 未超)。(待最後確認)

❓ 待確認 / Open questions

  • OQ1系統搜尋月費保單的供款期(CP)來源:spec 採用當前 quotation basic plan premium term。待確認:應否每張搜尋所得的保單採用自身的 premium term(有則用),無則 fallback 至當前 quotation term?
  • OQ2同一張月費保單,Monthly-only(原值)與 Mixed(× CP × 12)計法不同。暫假設 TRUE,待確認此差異為有意設計,且 Mixed 係與單筆上限比較(而非另一混合上限)。
  • OQ3超標行為未定:硬性攔截或警告後可繼續?是否有 escalation / Second-Checker / IO?錯誤訊息文字(中/英)及顯示位置?(demo verdict 暫用 placeholder)
  • OQ4邊界 / 失敗處理未定:CP 空/0/負/非整數;FX 匯率來源、日期、四捨五入、先轉抑或先加之次序;缺金額;同一保單同時手動加入及搜尋所得(去重);客戶搜尋失敗/離線;check 何時重跑。以上暫記 TBD。

此 demo 用作解釋計算邏輯,並非實際功能。數值為 sample,上限可編輯。命名沿用 project:「系統搜尋 / Auto-searched」對應 autoSearchedPolicy(demo 內亦稱「偵測 Detected」);「當前申請」= current application。
